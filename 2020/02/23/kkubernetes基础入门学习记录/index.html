各个组件说明

master节点

  • API server: 主节点上负责提供 Kubernetes API 服务的组件；它是 Kubernetes 控制面的前端。kube-apiserver 在设计上考虑了水平扩缩的需要。 换言之，通过部署多个实例可以实现扩缩

  • Scheduler: 根据策略调度，控制pod启动在哪个工作节点上，调度决策考虑的因素包括单个 Pod 和 Pod 集合的资源需求、硬件/软件/策略约束、亲和性和反亲和性规范、数据位置、工作负载间的干扰和最后时限

  • Control Manager: 主节点上运行控制器的组件，从逻辑上讲，每个控制器都是一个单独的进程，但是为了降低复杂性，它们都被编译到同一个可执行文件，并在一个进程中运行。

    这些控制器包括:

    • 节点控制器（Node Controller）: 负责在节点出现故障时进行通知和响应
    • 副本控制器（Replication Controller）: 负责为系统中的每个副本控制器对象维护正确数量的 Pod
    • 端点控制器（Endpoints Controller）: 填充端点(Endpoints)对象(即加入 Service 与 Pod)
    • 服务帐户和令牌控制器（Service Account & Token Controllers）: 为新的命名空间创建默认帐户和 API 访问令牌

  • ETCD: etcd is a etcd 是兼具一致性和高可用性的键值数据库，可以作为保存 Kubernetes 所有集群数据的后台数据库

node节点

  • kube-proxy: 集群中每个节点上运行的网络代理，实现kubernetes service 概念的一部分
  • kubelet: 部署在每一个工作节点，具有管理worker节点并负责和master通信,每个工作节点运行的代理组件，确保容器都创建在pod中，kubelet 接收一组通过各类机制提供给它的 PodSpecs，确保这些 PodSpecs 中描述的容器处于运行状态且健康。kubelet 不会管理不是由 Kubernetes 创建的容器

Pods/Nodes概述

Pods是k8s比较抽象的最小的工作单元， pod中可以包含多个container，多个container存在于pod中，pod中的容器共享同一个ip地址和端口空间(同一pod中的不同container端口不能相互冲突)

Service的概念

个人理解简单总结:
service就是对外公布的服务，告知外界的客户如何访问集群内部的服务，入口流量进入service，然后寻找当初service创建时选择关联的pod,这里的pod可以是单个pod也可以是多个pod
service需要创建独立的yaml文件，yaml文件中的kind类型选择service，并且通过标签选择器指令 LabelSelector 关联标签 也就是关联到pod pod创建的时候，可以指定标签， service创建后，就可以把流量分发到关联的pod(容器组)中

创建service时，对外暴露应用程序的集中方式,通过yaml文件type指令指定

ClusterIP(默认)
在群集中的内部IP上公布服务，这种方式的 Service（服务）只在集群内部可以访问到

NodePort
使用 NAT 在集群中每个的同一端口上公布服务。这种方式下，可以通过访问集群中任意节点+端口号的方式访问服务 :。此时 ClusterIP 的访问方式仍然可用。

LoadBalancer
在云环境中（需要云供应商可以支持）创建一个集群外部的负载均衡器，并为使用该负载均衡器的 IP 地址作为服务的访问地址。此时 ClusterIP 和 NodePort 的访问方式仍然可用

服务和标签

服务就是service，对外提供的服务，然后通过服务 关联包含标签的pod，从而负载到不同的pod中，有点像docker swarm的service概念，一个service多个副本，访问service的vip地址内部负载到副本中，在k8s使用 service关联不同的pod，来实现负载

service需要编写yaml文件，部署pod，编写service的时候通过spec.selector.app 指定标签，也就是会关联到 包含此标签的pod中去

标签就是在部署pod中，设置的自身标签，key+value

在业务环境中，例如微服务的架构，我们可以将pod指定 开发环境 生产环境 测试环境 准生产环境 设置不同类型标签，让service关联到具有相同作用的pod上

kubernetes的滚动更新说明

kubernetes的滚动更新，会启动一个新的pod，并分配新的ip，确保启动成功后，然后下线一个旧的pod，直到 和当初deployment部署的副本数量保持一致
